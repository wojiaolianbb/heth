## 执行规则
- 每次任务开始前必须阅读 AGENTS.md
- 只做当前 Phase，不提前实现后续功能
- 最小 diff，不修改无关文件
- 不引入 Brief 未明确允许的依赖
- 每个 Phase 完成后必须运行验证命令
- 每个 Phase 输出：Summary / Files changed / Verification / Risks
- 只修 blocking issues，不做额外重构

## 产品边界（硬规则）
- 本项目是个人自用的私人健康记录系统，不是公开健康内容站点
- 允许记录个人饮水、饮食照片、AI 营养估算、身高、体重、睡眠和运动数据
- 不要求面向公众的免责声明页面或内容详情页免责声明
- 禁止把产品描述为医疗诊断、治疗方案、疾病管理、药物建议或处方辅助工具
- 禁止出现：医疗诊断、治疗方案、药物名称、剂量建议、"适合 XX 疾病"

## 成本与服务边界（硬规则）
- 默认实现必须本地优先，并且不依赖强制付费服务才能启动核心记录功能
- 允许用户为个人用途显式配置自己的 AI provider、API key 或本地模型，用于饮食照片识别和营养估算
- 禁止把 API key、访问 token、个人照片或导出的健康数据提交到仓库
- 新增付费能力、云同步、远程数据库或第三方账号系统前，必须先有单独 Brief 和用户确认
- 动态数据优先使用本地文件、本地数据库、自托管服务或可免费运行的方案

## AI 分析边界（硬规则）
- AI 营养分析只能输出估算结果，不得输出医学结论
- 每次饮食照片分析必须保留：识别到的食物、份量假设、营养估算、可信度、需要用户确认的不确定项
- 必须允许用户修正食物名称、份量和营养估算结果
- 趋势复盘只能描述记录数据中的模式，例如饮水不足、睡眠时长波动、运动频率变化
- 禁止根据记录数据给出疾病判断、治疗建议、药物建议或剂量建议

## 数据边界（硬规则）
- 允许存储真实个人健康与生活记录：饮水、饮食照片、餐次、营养估算、身高、体重、睡眠、运动
- 个人数据默认本地保存；新增云同步前必须单独确认数据范围、加密策略、删除能力和备份策略
- 必须支持导入、导出和删除个人记录
- 所有动态数据必须有明确的数据模型、校验、错误状态和测试覆盖
- 日期记录 key 可以扩展，但每日汇总必须保持稳定、可迁移的日期维度

## UI 目标（硬规则）
- 必须同时支持手机端和 PC 端
- 手机端优先服务每日记录 loop：快速喝水、拍餐、补体重、补睡眠、补运动
- PC 端优先服务复盘和管理 loop：趋势分析、历史筛选、数据校正、导入导出和设置
- 任何新增页面必须考虑空状态、加载状态、错误状态和长内容溢出

## 技术栈
- Next.js 14 App Router + TypeScript + Tailwind CSS
- 测试：Vitest（或项目已有工具）
